# Preferences

- When writing any content, do not use em dashes. Use commas, periods, colons, semicolons, or parentheses instead.
- Never assume unclear requirements. Ask for clarification when anything is ambiguous, and before presenting solutions, research the latest available information relevant at that moment.
- For bigger tasks, break the work into small independent subtasks and run multiple parallel subagents when available, so output is faster and better.
- Non-negotiable: always write test cases first, then implement the feature or fix bug.
- Use every library and dependency to its full potential before adding another. A dependency must earn its weight: do not keep or add a library that is used for only a sliver of its purpose, or whose job a tool already in the project (or a few lines of our own code) can do. Before adding a library, check what is already installed and whether it covers the need; if an existing library is being underused or misused, fix that instead of layering on more. Periodically question whether each dependency still pays for the bundle size, runtime cost, and maintenance burden it adds. Example: an auth screen shipped a 2.1MB Lottie file plus the whole lottie-web/lottie-react runtime to render what turned out to be a single static raster (zero keyframes); the right fix was to extract the image to a ~128KB WebP and delete the library entirely.
- Before implementing major changes suggested by the user, critically evaluate if the suggestion makes sense, identify better alternatives, and assess technical feasibility. Present these findings and ask clarifying questions before proceeding.
- When the user asks to use a browser, use Google Chrome by default instead of Arc.
- Apply the Single Responsibility Principle when writing any code: each class, function, or module should have one reason to change. Split distinct concerns (for example, separate controllers/endpoints per audience or use case), put state and validation rules in their owning request/validator classes, and keep controllers thin orchestrators. Stay consistent with the existing codebase's conventions.
- Deploy ordering for changes that span frontend and backend: when **removing** a feature/endpoint, deploy the **frontend first** (remove the consumer), then the backend (remove the provider). This avoids a window where the live frontend still calls something the backend just deleted. Conversely, when **adding** a feature, deploy the **backend first** (provider exists before the consumer calls it). This bites whenever the frontend and backend ship on separate pipelines.
- All GitHub Actions workflows (especially deploys) must be **manual triggers only** (`workflow_dispatch`). Never wire `on: push` / `on: pull_request` / `on: schedule` auto-triggers for deploy or release pipelines. When creating or fixing a workflow, remove push/PR triggers and keep `workflow_dispatch`. Deploys are dispatched by hand (e.g. `gh workflow run <file> --ref <branch>`).
- When caching, queuing, or otherwise persisting a value that crosses process or request boundaries, store it as a plain array or JSON, not as a serialized class instance. Convert objects to arrays before storing (for example, `toArray()`) and rebuild them on read (for example, `fromArray()`). Serialized objects can come back as `__PHP_Incomplete_Class` (or fail to deserialize) when the reading process resolves classes differently, especially across a shared cache, separate serverless invocations, queues, or deploys. Example of the failure mode: cache a domain object in one request, read it back in another, and an `instanceof` check fails because it deserialized to `__PHP_Incomplete_Class`. Note that this class of bug often hides in tests, because the `array` cache driver stores objects in-memory without serializing, so always reason about the production store (Redis, SQS, etc.), not just the test driver.
